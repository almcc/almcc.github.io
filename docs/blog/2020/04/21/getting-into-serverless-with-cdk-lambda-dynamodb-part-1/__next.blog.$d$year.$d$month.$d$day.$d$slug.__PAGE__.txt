2:T4537,<p>AWS Cloud Development Kit (CDK) is a great tool for using code to deploy resources to AWS. In a series of posts, we will cover a few of the key concepts you need to be aware of getting started with AWS and CDK. We will create a Lambda Function that returns some data from a DynamoDB table over an API Gateway.</p>
<p>CDK allows you to define <a href="https://docs.aws.amazon.com/cdk/latest/guide/constructs.html">Constructs</a> in code that map to AWS components or resources, you can then collect those Constructs together into a <a href="https://docs.aws.amazon.com/cdk/latest/guide/stacks.html">Stack</a> which is what we deploy to an AWS region. Finally, a Stack is defined within an <a href="https://docs.aws.amazon.com/cdk/latest/guide/apps.html">App</a> which is the root of our project, it can contain many instances of the same stack for say multiple regions or multiple different stack instances.</p>
<p>You can choose to develop CDK projects with TypeScipt, JavaScript, Python Java or C#. While initially learning CDK I went for Python as it is usually my language of choice, I have later found TypeScript to be much more comfortable as there is a wealth of examples already out there. We will use TypeScript for this post.</p>
<p><strong>Note:</strong> Many tutorials and guides on learning AWS will suggest using the web console to setup and configure resources, however, beyond simple examples I have felt uncomfortable building so much in an unrepeatable fashion. If you are someone who learns by doing, then a pattern I have found works really well is building a setup partially with CDK, then using the web console to learn and experiment further before refactoring those changes back into the CDK stack itself. Simply rinse and repeat to develop ever bigger or more bespoke setups. Having AWS best practice built into the CDK constructs is a comforting safety net.</p>
<p>So that is already too many words, let's get started.</p>

The <code>aws configure list</code> command is a good way to test if you have your credentials set up correctly. For those of you who use profiles with the <code>-profile</code> flag, the <code>cdk</code> command works in the same way,</li>
</ul>

<h2>Step 3, The Lambda Function</h2>
<p>Install the lambda package.</p>
<pre><code class="hljs language-ruby">npm install <span class="hljs-variable">@aws</span>-cdk/aws-<span class="hljs-built_in">lambda</span>
</code></pre>
<p>Add the dependency to our stack (<code>lib/myapp-stack.ts</code>).</p>
<pre><code class="hljs language-tsx"><span class="hljs-keyword">import</span> * <span class="hljs-keyword">as</span> cdk <span class="hljs-keyword">from</span> <span class="hljs-string">"@aws-cdk/core"</span>;
<span class="hljs-keyword">import</span> * <span class="hljs-keyword">as</span> lambda <span class="hljs-keyword">from</span> <span class="hljs-string">"@aws-cdk/aws-lambda"</span>;

<span class="hljs-keyword">export</span> <span class="hljs-keyword">class</span> <span class="hljs-title class_">MyappStack</span> <span class="hljs-keyword">extends</span> <span class="hljs-title class_ inherited__">cdk.Stack</span> {
  <span class="hljs-title function_">constructor</span>(<span class="hljs-params"><span class="hljs-attr">scope</span>: cdk.<span class="hljs-title class_">Construct</span>, <span class="hljs-attr">id</span>: <span class="hljs-built_in">string</span>, <span class="hljs-attr">props</span>?: cdk.<span class="hljs-title class_">StackProps</span></span>) {
    <span class="hljs-variable language_">super</span>(scope, id, props);

    <span class="hljs-comment">// The code that defines your stack goes here</span>
  }
}
</code></pre>
<p>Next, we need to write some function lambda code. Create a file inside a newly created function directory (<code>function/index.py</code>). In this case, we are just going to return the top 3 finishers of the 2019 Tour de France, later we will work on returning their names from a DynamoDB table.</p>
<pre><code class="hljs language-python"><span class="hljs-keyword">import</span> logging
logger = logging.getLogger()
logger.setLevel(logging.INFO)

<span class="hljs-keyword">def</span> <span class="hljs-title function_">main</span>(<span class="hljs-params">event, context</span>):
    logger.info(<span class="hljs-string">"Start of function"</span>)
    <span class="hljs-keyword">return</span> {
        <span class="hljs-string">"statusCode"</span>: <span class="hljs-number">200</span>,
        <span class="hljs-string">"body"</span>: [<span class="hljs-string">"Egan Bernal"</span>, <span class="hljs-string">"Geraint Thomas"</span>, <span class="hljs-string">"Steven Kruijswijk"</span>],
        <span class="hljs-string">"headers"</span>: {
            <span class="hljs-string">"Content-Type"</span>: <span class="hljs-string">"application/json"</span>
        }
    }
</code></pre>
<p>Notice that I have also set up a logger with a simple logging statement, this will be useful later to demonstrate how we retrieve those logs.</p>
<p>Now we need to define our lambda function in our stack, <a href="https://docs.aws.amazon.com/cdk/api/latest/docs/@aws-cdk_aws-lambda.Function.html">we can use the <code>Function</code> construct from the <code>aws-lambda</code> package.</a> Inside the <code>MyappStack</code> construct in <code>lib/myapp-stack.ts</code> add the following code.</p>
<pre><code class="hljs language-tsx"><span class="hljs-keyword">const</span> myFunction = <span class="hljs-keyword">new</span> lambda.<span class="hljs-title class_">Function</span>(<span class="hljs-variable language_">this</span>, <span class="hljs-string">"MyFunction"</span>, {
  <span class="hljs-attr">runtime</span>: lambda.<span class="hljs-property">Runtime</span>.<span class="hljs-property">PYTHON_3_8</span>,
  <span class="hljs-attr">handler</span>: <span class="hljs-string">"index.main"</span>,
  <span class="hljs-attr">code</span>: lambda.<span class="hljs-property">Code</span>.<span class="hljs-title function_">fromAsset</span>(path.<span class="hljs-title function_">join</span>(__dirname, <span class="hljs-string">"../function"</span>)),
});
</code></pre>
<p>Don't forget to add <code>import * as path from "path";</code> as well.</p>
<p>You can see that I have defined a function imaginatively named <code>MyFunction</code>, it uses the Python 3.8 runtime, it's handler points to <code>main</code> function inside the <code>[index.py](http://index.py)</code> module and finally, the code lives in the <code>function</code> directory which is in the parent directory to our <code>lib</code> directory.</p>
<p>As it stands, this is enough for us to deploy our code. However, to invoke our function later we are going to need the function name that is created. CDK will take our <code>MyFunction</code> name and <a href="https://docs.aws.amazon.com/cdk/latest/guide/identifiers.html">add's to it to make sure that it's unique</a>. To extract that name of the function that will be created in AWS we can add a CloudFormation Output to our stack.</p>
<p>Again in <code>lib/myapp-stack.ts</code> and the following code under our lambda function declaration.</p>
<pre><code class="hljs language-tsx"><span class="hljs-keyword">new</span> cdk.<span class="hljs-title class_">CfnOutput</span>(<span class="hljs-variable language_">this</span>, <span class="hljs-string">"FunctionNameOutput"</span>, {
  <span class="hljs-attr">value</span>: myFunction.<span class="hljs-property">functionName</span>,
  <span class="hljs-attr">exportName</span>: <span class="hljs-string">"Lambda-Function-Name"</span>,
});
</code></pre>
<p>Note here that the <code>exportName</code> is the identifier for how we will reference the value and it <strong>must</strong> be unique across your entire AWS account, regardless of the stack. You can also see the reason we defined our function to a variable.</p>
<h2>Step 4, Build and Deploy</h2>
<p>Let's quickly check that our project builds.</p>
<pre><code class="hljs language-arduino">npm run build
</code></pre>
<p>We can also check that CDK can successfully convert (or synthesise) are code into a CloudFormation template.</p>
<pre><code class="hljs">cdk synth
</code></pre>
<p>I tend to pipe the output of <code>cdk synth</code> to less as output is long and it's useful to have a way to search it.</p>
<pre><code class="hljs">cdk synth | less

</code></pre>
<p>If all that worked then we are almost good to go. If this is the first time out have deployed a CDK project to your aws account you will need to <a href="https://docs.aws.amazon.com/cdk/latest/guide/tools.html">bootstrap you account</a>.</p>
<pre><code class="hljs">cdk bootstrap
</code></pre>
<p>Now we are ready to deploy our stack.</p>
<pre><code class="hljs">cdk deploy
</code></pre>
<p>You will be asked to approve any IAM policies and statements that CDK is creating. This is also a good insight into how policies are stitched together in AWS as well as a good reminder of the heavy lifting that CDK is doing for you.</p>
<p>When making changes to your project and deploying again, it can be useful to chain the build command and the deploy command together to avoid deploying a stale stack without your changes and then scratching your head as to why you change didn't work.</p>
<pre><code class="hljs language-arduino">npm run build &#x26;&#x26; cdk deploy
</code></pre>
<p>As I have used <code>&#x26;&#x26;</code>, <code>cdk deploy</code> will only run if the previous command (the build command) succeeds.</p>
