3:T2897,<p>I wrote recently about <a href="https://almcc.me/blog/2026/01/26/mcp-tools-vs-skills-with-scripts-why-simpler-might-be-better">MCP's patchy reliability</a> and how mature CLI tools wrapped in skills felt more stable. I wanted to explore that idea further — specifically, what if you could build custom Python CLI tools with a skills wrapper and have the same success without managing virtual environments?</p>
<p>Python-tool-skills are that exploration. They're Python CLI tools built with Click, run via <code>uvx</code>, and bundled inside skills. No venv pollution, locked dependencies, just works.</p>
<h2>What works: wrapping mature CLI tools</h2>
<p>Most CLI tools are battle-tested, predictable, well-documented. Wrapping them in skills works brilliantly. The agent calls a command, parses the output, moves on.</p>
<p>These tools have been used by thousands of developers across wildly different contexts. Error messages are clear. Edge cases are handled. You're building on solid ground.</p>
<p>Most importantly, they are a text efficient, well understood textual interface — one well suited to LLMs. With in-context <code>--help</code> output, the agent can discover capabilities progressively rather than loading full schemas upfront.</p>
<p>But what if you need a custom tool? Skills today tend towards writing a bash script, but anything beyond basic text parsing gets messy. Python's cleaner, but then you're managing virtual environments, dependencies, import paths.</p>
<p>Enter uvx — a tool runner that handles Python package isolation automatically, no venv required. Well, not one you have to manage yourself at least!</p>
<h2>Python-tool-skills: CLI tools bundled in skills</h2>
<p>So why don't we try building custom tools the same way — as standalone CLIs that just happen to be written in Python?</p>
<p>I've put together what I'm calling a python-tool-skill — an AI agent tool packaged as a skill that bundles a Python CLI in its <code>assets/</code> directory. The CLI is built with <a href="https://click.palletsprojects.com/">Click</a> and runs via <a href="https://docs.astral.sh/uv/guides/tools/"><code>uvx</code></a>.</p>
<p>You can see examples and try it yourself at <a href="https://github.com/almcc/python-tool-skill-creator">github.com/almcc/python-tool-skill-creator</a>.</p>
<pre><code class="hljs language-objectivec">f1/
├── <span class="hljs-built_in">SKILL</span>.md          ← trigger description + agent instructions
└── assets/
    ├── f1.py         ← Click <span class="hljs-built_in">CLI</span> entry point
    ├── pyproject.toml
    └── uv.lock
</code></pre>
<p>The agent runs it with:</p>
<pre><code class="hljs language-bash">uvx --from /path/to/skill/assets f1 races
</code></pre>
<p>No venv, no <code>__pycache__</code>, no <code>.pyc</code> files. <code>uvx</code> handles isolation automatically. Dependencies are pinned in a committed <code>uv.lock</code> — reproducible and offline-capable after first cache population.</p>
<p>(I chose F1 as a subject because we are very excited for the 2026 season in our house!)</p>

<h2>Why uvx</h2>
<p><a href="https://docs.astral.sh/uv/">uv</a> is a fast Python package manager. <code>uvx</code> is its tool runner — like <code>npx</code> for Python.</p>
<p>Point <code>uvx</code> at a directory with a <code>pyproject.toml</code>, and it installs the package in an isolated environment and runs it. The environment is cached. Subsequent calls are fast.</p>
<pre><code class="hljs language-bash"><span class="hljs-comment"># First run: installs dependencies, runs command</span>
uvx --from /path/to/assets f1 races

<span class="hljs-comment"># Subsequent runs: uses cached environment</span>
uvx --from /path/to/assets f1 race 5

<span class="hljs-comment"># After editing f1.py: force reinstall</span>
uvx --from /path/to/assets --reinstall f1 races
</code></pre>
<p>No <code>pip install</code>, no <code>python -m venv</code>, no activation scripts. It just works.</p>

<h2>Why this works for me</h2>
<p>I've been using python-tool-skills for a week or so and they feel significantly more user-friendly than creating custom MCP servers. The biggest win? The tools are just CLI programs. I can run them directly from my terminal, test them independently, and use them outside the agent context.</p>
<h2>Comparing to MCP servers</h2>
<p>MCP servers offer fine-grained control — you can enable and disable specific tools, control exactly what the agent sees. With CLI-based skills, you lose that granularity. You load the skill, the agent gets access to everything the CLI can do.</p>
<p>But as we trust LLMs more, is this much of a concern? The underlying authentication model of the tool you're calling might be enough. If the tool requires credentials, the agent still can't do anything you haven't authorized. The permission boundary shifts from the agent runtime to the tool itself.</p>
<p>Of course, the LLM still needs a way to instigate a tool call. That mechanism may be MCP, or some kind of custom tool schema specific to that agent runtime. The proposal here isn't about replacing those mechanisms — it's about shifting the interface at which we (the community) build our tooling. Rather than building custom protocols and servers, we build portable CLI tools that any agent can wrap.</p>
<p>I do wonder if we'll eventually see a shell dedicated for LLMs — a safer sandbox in which they can work. Something that provides better guardrails, audit logging, and resource limits than a standard shell, while maintaining the familiar CLI interface pattern.</p>
<h2>The cost advantage</h2>
<p>There's another compelling reason to prefer CLI tools: they're dramatically cheaper to use. As <a href="https://kanyilmaz.me/2026/02/23/cli-vs-mcp.html">documented by Kan Yilmaz</a>, CLI-based tools use ~94% fewer tokens than equivalent MCP servers through progressive disclosure — the agent only loads tool details when it needs them via <code>--help</code>, rather than loading complete JSON schemas for every tool upfront.</p>
<p>With 84 tools loaded, MCP burns ~15,540 tokens at session start. The same tools via CLI? ~300 tokens. The agent discovers what it needs on-demand, keeping context focused and costs low.</p>
<h2>Beyond Python</h2>
<p>Of course, this pattern isn't limited to Python. You could bundle Ruby tools with <code>gem install</code>, Node.js tools with <code>npx</code>, Go binaries, Rust crates — anything that can run as a CLI and be distributed in a portable way. Python-tool-skills just happen to be the convenient starting point I chose.</p>
<h2>The future of agent tooling</h2>
<p>Sentry clearly agrees that CLI tools are the way forward — they've published <a href="https://docs.sentry.io/ai/sentry-cli/">official agent tooling documentation</a> centered around their CLI, not MCP. Will we see skills become a new distribution method for application tooling? Package your CLI, add usage instructions, ship it as a skill. Developers get tools they can use directly. Agents get tools they can discover progressively.
