2:T6e0,<p>Read the full writeup: <a href="https://almcc.me/blog/2026/03/02/an-alternative-to-mcp-python-cli-based-agent-skills">An Alternative to MCP - Python CLI-Based Agent Skills</a></p>
<h2>What it is</h2>
<p>A python-tool-skill packages a Python CLI tool in a skill's <code>assets/</code> directory. The tool is built with <a href="https://click.palletsprojects.com/">Click</a> and run via <a href="https://docs.astral.sh/uv/guides/tools/"><code>uvx</code></a>, which handles environment isolation automatically.</p>
<pre><code>skill-name/
├── SKILL.md          ← trigger description + agent instructions
└── assets/
    ├── tool.py       ← Click CLI entry point
    ├── pyproject.toml
    └── uv.lock
</code></pre>
<p>The agent invokes it like any other CLI tool:</p>
<pre><code class="language-bash">uvx --from /path/to/skill/assets tool-name &#x3C;command>
</code></pre>
<h2>Why it exists</h2>
<p>CLI tools provide text-efficient interfaces that serve as excellent agent tools. Too often, custom skills devolve into messy bash scripts that exceed their original purpose. Python Click CLIs offer structure and a solid foundation for building upon.</p>
<p>Python-tool-skills let you build custom tools with the same simplicity as wrapping existing CLIs. uvx handles environment isolation, dependencies are locked in a committed <code>uv.lock</code>, and there's no repo pollution.</p>
<h2>Key benefits</h2>
<ul>
<li><strong>Just CLI programs</strong>: Test independently, use outside agent context</li>
<li><strong>Fewer tokens</strong> than MCP through progressive disclosure (agent loads <code>--help</code> on demand rather than full schemas upfront)</li>
<li><strong>No server startup process</strong> to manage</li>
</ul>
